# 設計說明：JRF OCR 管線的架構與取捨

本文件解釋這個系統是怎麼想出來的，以及各個設計決策背後的理由。  
這不是操作手冊——操作請見 [HOW-TO-DRIVE.md](HOW-TO-DRIVE.md)，環境建置請見 [SETUP.md](SETUP.md)。

---

## 為什麼用 AI 做 OCR，而不用傳統工具

Tesseract 或 Adobe Acrobat 內建的 OCR 對中文法院卷宗成效有限，原因有幾個：

- **印章與戳記**：法院文件常有旋轉、疊印、褪色的橡皮章，傳統 OCR 看到的是雜訊，Claude 可以辨識章文並理解它代表什麼。
- **混合排版**：同一頁可能有直排標題、橫排正文、手寫註記、表格。傳統 OCR 依賴固定的版面假設，AI 視覺模型沒有這個限制。
- **字形混淆**：「己」與「已」、「戊」與「戍」在低解析度掃描下極難辨別，Claude 根據上下文判斷，而不只是比對像素。
- **掃描品質不一**：不同法院、不同年代的卷宗掃描品質差異很大。AI 模型對模糊、偏斜、光線不均的頁面仍有一定容錯能力。

代價是成本與速度：每頁圖片都消耗 token，且必須透過 API（或 Claude Code Max Plan）。這個選擇是在準確度與成本之間做的取捨，對法律文件而言準確度優先。

## 為什麼把 OCR 切成多個步驟，而不是一次完成

整個管線分成六個獨立步驟（轉圖、OCR、QA、分段、標記、書籤），原因有三：

**1. 認知任務不同。** 「讀圖寫字」和「判斷文件邊界」和「萃取元資料」是本質上不同的任務，用同一個 prompt 一次要求 AI 完成全部，準確度會下降。每個步驟有自己專屬的 prompt，可以針對性地調整。

**2. 失敗可以局部恢復。** 一個 200 頁的 PDF 如果在標記階段失敗，不需要從頭重新 OCR。`state.json` 記錄每個 chunk 的進度，下次從斷點繼續。

**3. 人工可以介入。** 分段步驟（Step 3）輸出一個 prompt，讓操作者貼入 Claude Code 取得 JSON。這個刻意的「人工關卡」讓有疑問的文件邊界可以被人工確認，而不是完全自動化然後悄悄出錯。

---

## 為什麼 OCR 每次只讀 3 頁，而不是一次讀整個 chunk

context window 的限制。一張 200 DPI 的 A4 頁面圖片約佔 1,000–2,000 token，讀取 50 頁圖片本身就消耗 5 萬–10 萬 token，再加上 OCR 輸出文字，一個 session 的 context 很快就會用盡。

每次派出一個 subagent 讀 3 頁，subagent 完成任務後退出，不保留狀態。主 agent 只負責協調，不讀圖片，所以 context 維持在可控範圍。50 頁的 chunk 分成 17 個 batch，每個 batch 獨立、可以平行跑。

這也是為什麼 QA 步驟在所有 batch 完成後才執行——QA 需要看整個 chunk 的 OCR 輸出，但那時 OCR subagent 都已退出，context 是乾淨的。

---

## 為什麼有獨立的 QA 步驟

OCR 錯誤有一個重要特性：它們是**系統性的**。同一個字在同一台掃描機、同一種字體下，幾乎每次都會被誤辨成同一個錯字。這意味著錯誤可以被歸納成規則。

`tools/correction_rules.yaml` 儲存這些規則，隨著處理案件的累積不斷增加。QA subagent 套用這些規則做自動修正，同時把新發現的可疑錯誤寫入 `qa_log.jsonl`，供人工審閱後決定是否升格為規則。

如果把校正邏輯混入 OCR prompt，每次改規則都要同時測試 OCR 效果，兩件事互相干擾。獨立的 QA 步驟讓兩者可以分開迭代。

## 為什麼分段（segments）和標記（tagged）是兩個不同步驟

**分段**（Step 3）回答的問題是：這份 OCR 文字裡有幾份文件？每份從哪頁到哪頁？

**標記**（Step 4）回答的問題是：每份文件是什麼類型？書狀？裁定書？送達證書？日期是幾月幾號？當事人是誰？

這是兩個不同的認知任務。分段需要看整個 chunk 的文字流，判斷轉折點；標記需要深入閱讀每份文件的內容。把它們合併成一個 prompt，往往讓模型在兩者之間顧此失彼。分開之後，每個 prompt 可以針對自己的任務給出更清晰的指示，輸出也更容易驗證。

## 為什麼 Drive 整合是「包在外面」的，而不是嵌入管線

OCR 管線（Steps 1–6）完全不知道 Google Drive 的存在，它只處理本機檔案路徑。Drive 整合是一個獨立的 CLI（`drive_pipeline.py`），在管線的前後各執行一次：

```
Drive 下載 → [本機 OCR 管線] → Drive 上傳
```

這個設計的好處：
- 沒有 Drive 設定也可以跑 OCR（直接用本機 PDF）
- 管線可以獨立測試，不需要模擬 Drive API
- Drive 層的邏輯（OAuth、manifest、重試）不會污染 OCR 工具的程式碼

代價是操作者需要手動串接兩邊——這是刻意的取捨，清晰優先於方便。

---

## 為什麼用 Google Sheet 作為協調層

Queue 頁籤和 Archive 頁籤讓非技術背景的使用者（律師、法務助理）可以：

- 直接在 Google Drive 上傳 PDF，不需要打指令
- 在試算表看到每個案件的處理狀態
- 在 Archive 頁籤搜尋、篩選歸檔記錄

試算表是非技術人員熟悉的介面。把 Queue 設計成 Google Sheet 的一個頁籤，而不是資料庫或 JSON 檔，是一個刻意降低使用門檻的決定。

---

## 為什麼有兩個狀態檔（state.json 和 drive_manifest.json）

`state.json` 追蹤 **OCR 管線的進度**：哪些 chunk 已完成、每個步驟的狀態。它是給 Claude Code 看的，格式圍繞著 chunk 和 step 設計。

`drive_manifest.json` 追蹤 **Drive 上傳的狀態**：哪些檔案已上傳、Drive URL 是什麼、哪些 CSV 行已寫入試算表。它是給 `drive_pipeline.py` 看的，格式圍繞著 upload entry 設計。

兩者的生命週期不同：OCR 管線可能跑好幾天（大型案件），Drive 上傳通常是管線結束後的一次性動作。把它們合在一個檔案會讓兩種邏輯的程式碼互相糾纏；分開之後，各自只關心自己的狀態機。
